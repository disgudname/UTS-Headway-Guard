Bus Arrival Feeds

Real-time bus arrival predictions for any stop, in two formats: CAP 1.2 for alerting platforms such as Rave and Alertus, and RSS 2.0 for digital signage devices such as NovaNex LED displays.

Step 1: Create a Feed Code

Do this first. TransLoc stop IDs can change without notice, and sometimes represent one physical stop as separate IDs per route. Both feed formats below use a short, stable code instead of a raw stop ID, so ops staff can repoint a code to a new stop ID later without touching the signage device or redeploying the dashboard.
1 Go to uva.transloc.com and click the stop you need on the map.
2 The stop ID appears in the URL, e.g. https://uva.transloc.com/stops/26 → stop ID 26. A code can map to more than one stop ID if TransLoc splits one physical stop into per-route IDs — arrivals from every mapped ID are merged into a single feed.
3 Go to /feed-codes (dispatcher login required) and add a short code (e.g. NGPG for North Grounds Parking Garage, JPA for Jefferson Park Ave @ West Complex), the stop ID(s) from step 2, and an optional label.
4 Use that code in the CAP or RSS feed URL for your device instead of a raw stop ID. If TransLoc renumbers the stop later, repoint the code at /feed-codes and the signage URL never has to change.

Don't have dispatcher login access to create a code yourself? Contact Pat Cox (phc6j) and a feed can be set up for your stop.

CAP Feeds

Feed URL

https://utsopsdashboard.com/api/cap/stop_arrivals/{code}

Replace {code} with a code from Step 1. Updates every 15 seconds server-side; poll at least once per minute. Raw stop IDs still work too: /api/cap/stop_arrivals?stopID={stopID}, but prefer the code-based URL for anything newly deployed.

Example

Feed for Massie Rd @ JPJ South Lot (stop 26):

https://utsopsdashboard.com/api/cap/stop_arrivals?stopID=26
<?xml version="1.0" encoding="UTF-8"?>
<alert xmlns="urn:oasis:names:tc:emergency:cap:1.2">
  <identifier>uts-stop-26-20260305T1430Z</identifier>
  <sender>utsopsdashboard.com</sender>
  <sent>2026-03-05T14:30:00+00:00</sent>
  <status>Actual</status>
  <msgType>Alert</msgType>
  <scope>Public</scope>
  <info>
    <language>en-US</language>
    <category>Transport</category>
    <event>Bus Arrivals - Massie Rd @ JPJ South Lot</event>
    <urgency>Expected</urgency>
    <severity>Minor</severity>
    <certainty>Observed</certainty>
    <effective>2026-03-05T14:30:00+00:00</effective>
    <expires>2026-03-05T14:32:00+00:00</expires>
    <senderName>UVA Transit</senderName>
    <headline>Silver:0m,8m Gold:3m,10m</headline>
    <description>Silver:0m,8m Gold:3m,10m</description>
    <web>https://utsopsdashboard.com/arrivalsdisplay?stopid=26</web>
  </info>
</alert>

Feed Structure

FieldDescription
<identifier>Stays the same across polls as long as the displayed text hasn't changed; only changes when arrivals/alert content actually updates, so a CAP client doesn't treat every poll as a brand-new alert
<effective>10 seconds before generation time — guarantees it's already in the past by the time a polling client checks it
<expires>Two minutes after generation time, refreshed on every poll — never goes stale as long as the feed keeps being polled
<headline> / <description>All routes and next two arrivals, e.g. "Silver:0m,8m Gold:3m,10m" — space-separated between routes, comma-separated (no space) between times, "0m" instead of "Due" for an arriving bus, trailing "Line"/"Loop" dropped from route names
<web>Link to the live arrivals display for the stop

One <info> block per feed, all routes combined into a single message, sorted by soonest arrival ("No buses currently scheduled" if none). If an active service alert applies, its text is appended after the arrivals — the sign paginates in fixed 2-line pages and only the first page reliably starts at line 0, so arrivals (the priority content) go first and the alert absorbs whatever page-parity is left over.

Why the compact "Route:time,time" format? The sign's word-wrap is pixel-width-based and only breaks on spaces, so each route is packed into one space-free token to guarantee it's never split mid-route across a line break — routes are then joined with a single space. A route's second ETA is dropped whenever the first is already double-digit minutes, since two double-digit times back to back risks overflowing a line.

RSS Feeds

Feed URL

https://utsopsdashboard.com/api/rss/stop_arrivals/{code}

Same codes as CAP. Raw stop IDs still work too: /api/rss/stop_arrivals?stopID={stopID}, but prefer the code-based URL for anything newly deployed.

Example

Feed for Massie Rd @ JPJ South Lot (stop 26):

https://utsopsdashboard.com/api/rss/stop_arrivals?stopID=26
<?xml version="1.0" encoding="UTF-8"?>
<rss version="2.0">
  <channel>
    <title>Bus Arrivals - Massie Rd @ JPJ South Lot</title>
    <ttl>1</ttl>
    <item>
      <title>Silver:0m</title>
      <description>Next arrivals: 0m, 8m</description>
    </item>
    <item>
      <title>Gold:3m</title>
      <description>Next arrivals: 3m, 10m, 18m, 24m</description>
    </item>
  </channel>
</rss>

Feed Structure

FieldDescription
<title> (channel)Stop name, e.g. "Bus Arrivals - Massie Rd @ JPJ South Lot"
<ttl>1 (minute) — recommended poll interval
<item> <title>Route name and next arrival, e.g. "Gold:3m" or "Silver:0m" — same compact format as CAP's headline (see above)
<item> <description>All upcoming arrivals for that route, e.g. "Next arrivals: 3m, 10m, 18m"

One item per route, sorted by soonest arrival (a single "no arrivals" item if none are scheduled). If an active service alert applies, its text leads the feed, ahead of the arrivals — the opposite placement from CAP, since RSS clients aren't known to have the same fixed-page pagination the OnAlert sign does.

Service Alerts

Active system notices (created by dispatch at /admin) targeting a stop, or set to apply system-wide, show up automatically in that stop's feed using their short-form text (falling back to the full home page message if none was set). Notice severity does not change the CAP feed's severity/urgency fields (always Minor/Expected) — this is routine bus-arrival signage and should never be mistaken for the university's actual emergency feed.
